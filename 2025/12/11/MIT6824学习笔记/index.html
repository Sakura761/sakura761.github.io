Google MapReduce 学习笔记

1. 核心理念与背景

  • 背景问题：Google 内部需要处理海量的原始数据（如网页爬虫数据、日志），涉及大量的计算（如倒排索引构建、文档聚类）。虽然核心运算很简单，但为了在短时间内完成，必须将计算分布到成百上千台机器上。
  • 核心痛点：分布式计算带来了极大复杂性，包括并行计算、数据分发、错误处理（容错）、负载均衡等。
  • MapReduce 的诞生：为了将“业务逻辑”与“分布式系统的复杂性”剥离，Google 设计了 MapReduce 抽象模型。程序员只需要编写简单的 Map 和 Reduce 函数，系统自动处理所有底层的分布式细节。

2. 编程模型 (Programming Model)

MapReduce 基于函数式编程语言（如 Lisp）中的原语，核心由两个函数组成：

  1. Map 函数：
    • 输入：一对 key/value。
    • 处理：产生中间 key/value pair 的集合。
    • 逻辑：Map(k1, v1) -> list(k2, v2)
  2. Reduce 函数：
    • 输入：中间 key (k2) 和它对应的一组 value 集合。
    • 处理：合并这些 value，产生较小的一组 value（通常是 0 个或 1 个）。
    • 逻辑：Reduce(k2, list(v2)) -> list(v2)

经典案例：单词统计 (Word Count)

  • Map: 读取文档，对每个词输出 <word, 1>。
  • Reduce: 将相同 word 的所有 1 加起来，输出 <word, total_count>。

3. 系统架构与执行流程

MapReduce 运行在由普通商用 PC 组成的大型集群上，依赖 GFS（Google File System）存储数据。

3.1 角色分工

  • Master (1个)：管理者。负责分配任务、监控 Worker 状态、管理中间文件位置。
  • Worker (多个)：干活的。由 Master 分配执行 Map 任务或 Reduce 任务。

3.2 执行流程 (Execution Overview)

  1. 输入分片 (Split)：将输入文件切分为 M 个分片（通常 16MB-64MB）。
  2. 任务分配：Master 将 M 个 Map 任务和 R 个 Reduce 任务分配给空闲的 Worker。
  3. Map 阶段：
    • Map Worker 读取对应分片内容。
    • 执行 Map 函数，结果缓存在内存。
    • 定期将缓存写入本地磁盘，并根据分区函数（如 hash(key) mod R）切分成 R 个区域。
    • 关键点：Map 输出在本地磁盘，而不是 GFS。位置信息汇报给 Master。
  4. Shuffle 阶段 (数据传输)：
    • Master 通知 Reduce Worker 数据的位置。
    • Reduce Worker 通过 RPC 从 Map Worker 的磁盘读取数据。
    • Reduce Worker 对读取到的数据按 key 进行排序（使得相同 key 的数据聚在一起）。
  5. Reduce 阶段：
    • Reduce Worker 遍历排序后的数据，对每个唯一 key 调用 Reduce 函数。
    • 输出结果追加到最终的输出文件中（存储在 GFS）。
  6. 完成：所有任务完成后，Master 唤醒用户程序。

4. 关键设计特性 (核心竞争力)

4.1 容错机制 (Fault Tolerance)

由于集群规模巨大，机器故障是常态。

  • Worker 故障：
    • Master 周期性 Ping Worker，超时则标记失效。
    • Map 任务：即使已完成，如果 Worker 挂了，也需要重做（因为结果存在挂掉机器的本地磁盘上，无法访问）。
    • Reduce 任务：如果已完成，不需要重做（结果存在全局文件系统 GFS 上）。
  • Master 故障：
    • 使用 Checkpoint（检查点）机制写入磁盘。如果 Master 挂了，通常选择终止运算，由用户决定是否重试。

4.2 数据本地化 (Locality)

  • 带宽是稀缺资源。
  • Master 在调度 Map 任务时，会尽量将任务分配到存储了该数据分片的机器上（或附近的机器）。
  • 结果：绝大多数 Map 读取操作是在本地磁盘进行的，极大节省了网络带宽。

4.3 备用任务 (Backup Tasks) - 解决“长尾效应”

  • 落伍者 (Straggler)：如果一台机器因为硬盘坏了或负载高，导致最后几个任务执行极慢，会拖累整个作业的完成时间。
  • 机制：当操作接近完成时，Master 会为剩下的正在执行的任务启动备用副本。无论是主副本还是备用副本先做完，都算任务完成。
  • 效果：稍微多占用一点资源，但能显著减少大型作业的总执行时间（论文提到排序任务快了 44%）。

5. 实用技巧 (Refinements)

  1. Partitioning (分区函数)：默认是 Hash，但用户可以自定义（例如按主机名分区，让同一主机的 URL 进同一个文件）。
  2. Combiner 函数：
    • 在 Map 端先进行一次局部的 Reduce。
    • 例如 Word Count，Map 输出 <the, 1> 几千次。通过 Combiner，先在本地合并成 <the, 3000> 再通过网络发送给 Reduce。
    • 大幅减少网络传输量。
  3. Skipping Bad Records：如果某条特定的记录总是导致代码 Crash，系统可以检测到并选择跳过该记录，保证整体作业能完成。
  4. Local Execution：提供本地模拟执行模式，方便调试（gdb）。
  5. Status Information：Master 提供 HTTP 页面，显示进度、状态、错误日志等。

6. 性能与经验

  • 线性扩展性：机器越多，处理速度越快（在一定范围内）。
  • 实际应用：Google 用 MapReduce 重写了索引系统。代码量从 3800 行 C++ 减少到 700 行，且更易于维护和容错。
  • MapReduce 的成功原因：
    1. 封装了复杂的底层细节（并行、容错、数据传输）。
    2. 适用于大量不同类型的问题。
    3. 能在廉价的普通机器集群上高效运行。

7. 总结

MapReduce 不仅是一个计算模型，更是一个运行在不可靠硬件上的高可靠分布式系统架构。它通过移动计算而非移动数据（Data Locality）、自动容错（Re-execution）和备用任务（Backup Tasks）三大法宝，成功解决了大规模数据处理的难题。
